Kurento Room v6.6.1 不完全翻译

介绍

Kurento致力于为Web和智能手机平台的高级视频应用开发人员提供服务。发现一个常用的场景是：需要媒体服务器进行高级媒体处理的多人通话。
这个项目介绍了Rooms API，专门用于开发以Kurento媒体服务器为核心的会议应用程序。会议组（会议室）将使用API进行管理，同时还提供了通过Kurento Media Server实例互连最终用户的方法。

API的核心模块Room SDK是用于服务器端的Java库，具有管理多会议会话的主要功能。
此外，还提供额外的组件，可用于开发遵循上述架构的应用程序：

  • Room Serve：基于容器实现的服务器，通过WebSockets使用JSON-RPC与客户进行通信
  • Room JavaScript Client：为Web应用程序实现一个Room间客户端的模块（仅与Room Server一起使用）
  • Room Client：用于Java Web应用程序或Android客户端的客户端库（仅与Room Server一起使用）

Core API

核心模块为使用Java语言开发的应用程序定义了一个编程模型。使用这个库的一些好处：

  • 控制多媒体会议（room）的生命周期
  • 管理参与者所需的典型操作（加入，退出，发布或接收媒体流等）
  • 简单的媒体信令，应用仅需要提供从客户端接收的媒体初始化和协商参数
  • 多媒体分支或服务拓扑默认是隐藏的（像图像过滤器这样的媒体元素仍然可以应用于参与者的流）

该组件需要访问至少一个Kurento媒体服务器实例以进行WebRTC媒体处理。
开发人员可以将这个API直接集成到他们的应用程序中，但是有一点需要认识到，这个库本身不具备与客户端通信的能力。更多细节可以在核心API部分找到。

其它组件

服务器实现，即房间服务器，使用Springboot、基于WebSockets协议的JSON-RPC打包了Room SDK的功能。因此，它结合了客户端通信组件和核心Room API，提供了一个完整Rooms server应用程序。它可以很容易地集成到使用Spring Boot框架的其他应用程序中。这两个客户端库都被设计为与Room Server结合使用，因为它们使用服务器组件公开的JSON-RPC API。
Java客户端不处理媒体操作，而只是通过Room Server支持的JSON-RPC协议来定义Java API。
另一方面，JavaScript客户端还控制浏览器的媒体初始化和协商细节，使开发人员可以专注于应用程序功能。
目前，还没有可用的IOS客户端。
请注意服务器的WebSocket API对用户名称有一个限制，它不能包含下划线（_）。

需要注意的是，尽管核心模块是Room SDK，开发人员可以自由选择他们需要的任何组件来构建他们的应用程序。但是，有些可能需要在客户端进行一些小的修改，而另外一些则希望将其逻辑注入到服务器端，甚至不得不修改SDK。例如：demo演示了如何将一些提供的组件与客户端技术AngularJS以及对服务器的一些小修改（为视频流和自定义消息类型添加图像过滤器）集成在一起。

功能

在启动并运行SpringBoot框架后，它启动模块kurento-room-server的主应用程序，通过一个WebSocket接口公开房间API。

Rooms核心API

Rooms API基于Room Manager抽象。Room Manager可以借助Kurento技术来组织和控制多方的群组通话。
我们将此库理解为任何想要实现Room服务端应用程序的开发人员的SDK。
Room Manager的Java API负责处理Room和媒体特定的细节，使程序员从低级或重复性任务（每个多重会议应用程序固有的）中解放出来，让他们更专注于应用程序的功能或业务逻辑。

理解API

Manager处理了两个主要概念：

  • rooms：虚拟的同伴组，限制了用户一次只能加入一个组。为了识别用户，使用他们的名字。
  • participants：虚拟的终端用户。应用程序将提供用户级别的字符串表示形式，以便唯一标识此参与者

鉴于使用了我们API的应用程序的性质，期望终端用户能够尝试连接到现有房间（或创建新的房间），并发布或接收来自其他对等体的媒体流。
在使用这个SDK时，应用程序的工作是从终端用户那里接收消息并将其转换向Room Manager实例的请求。
API的一些方法不仅要处理房间管理，而且要处理参与者所需的媒体功能。底层媒体处理是通过一个名为Kurento Client的库来执行的，当一些由管理器创建的媒体对象满足某些条件时，可以出发事件。反过来，处理这些事件所收集的信息有时需要通知终端用户。Manager通过Room Handler接口来通知应用程序最重要的事件，应用程序必须提供一个实现。

提供了两种类型的Room Manager，它们暴露了相同的接口（相同的方法名称，但具有不同的签名）：

  • org.kurento.room.RoomManager：默认实现
  • org.kurento.room.NotificationRoomManager：定义了发送通知或响应客户端的模型的实现

RoomManager

创建新的Room Manager(普通)有两个要求，需要提供如下实现：

  • Room Handler责由内部媒体对象触发的事件
  • Kurento Client Manager用于获取Kurento客户端示例

对于来自客户端的请求，应用程序需要通知远程参与者执行请求的结果，例如当其中一个请求发布她的媒体时通知房间中的所有参与者。
还有另外一种类型的方法来处理由应用程序发起的请求（或业务逻辑），在这种情况下，应用程序可以自由解释结果并对其执行操作。

Events - RoomHandler

为了向一个媒体对象发出的事件，如收集新的ICE候选人或媒体错误，应用程序必须提供一个事件处理机制。一般来说，这些都是面向用户的事件，所以应用程序应该通知相应的用户。

NotificationRoomManager

实例化NotificationRoomManager时有两个要求，需要提供如下实现：

  • 能够发送消息或通知连接到应用程序的终端用户的接口或者控制通知的生命周期的房间处理通知
  • Kurento Client Manager用于获取Kurento客户端示例

房间事件处理程序的设计旨在向应用程序提供处理用户请求结果的反馈。
通知管理API考虑了两种不同类型的方法：

  • server domain - 包括用于实现应用程序逻辑层的方法以及与Room SDK的集成。这些方法将以同步的形式执行。它们可以被视为帮助或管理方法，并直接控制Rooms。
  • client domain - 由于传入的用户请求而调用，它们实现了客户端的房间规范。它们可以异步执行，调用方不应期望得到结果，但是如果需要进一步分析和处理客户端的请求，可以使用response handler。

图描述了在使用notifications room manager时的系统组成组件：

Notifications design - UserNotificationService

这个规范是设计好的，这样房间管理员可以将通知或响应发送回远程对等点，同时保持与传输层或通信层的隔离。通知API被NotificationRoomHandler的实现类(DefaultNotificationRoomHandler类)使用。
API的方法是基于协议json - rpc及其消息格式定义的。希望客户端和服务器通信使用此协议，但不是必需的。
开发人员需要为这个API提供一个实现。
如果开发人员选择另一种机制与客户端进行通信，那么他们将不得不使用他们自己的NotificationRoomHandler的实现，它将和来自Room API的通信细节解耦。

Notifications design - NotificationRoomHandler

通过这个接口，Room API将客户端原语的执行结果传递给应用程序，并从那里传递给客户端。遵守这样的协议是应用程序的职责。这些方法都返回void。
在用户请求范围之外发生的事情会调用一些方法:房间关闭、用户被逐出，以及从RoomHandler接口继承的那些。

KurentoClientProvider

设计该服务的目的是房间管理员可以随时获取一个Kurento客户端实例，而不需要了解KMS实例的位置。
开发人员需要为这个接口提供一个实现。

POJOs

在房间API定义的请求和响应中使用了以下类：

  • UserParticipant - 将参与者的标识符与用户名联系起来，并标记该用户是否是当前流媒体。
  • ParticipantRequest - 将参与者的标识符与请求ID(通信级别的可选请求标识符，包含客户端响应；是可以为null的，并且将被复制)联系起来。在Room Manager的通知变体中使用。
  • RoomException - 运行时异常包装，包括：
    • code - 表示发生的错误类型的数字
    • message - 对错误的简短字符串描述
